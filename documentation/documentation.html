1.Introduction

1.1 Purpose

The purpose of this web application is to allow users to acces a plethora of different resources regarding "Creative Coding". "Creative Coding" focuses on form rather than function and allows programmers to access their creative side. The app will act similarly to a search engine or a collection of multiple links to different resources from the web.

2. Overall Description

2.1 Product Perspective

CroG is a self-contained product, represented by a web application with an extensive user interface.

2.2 Product Functions

The application will allow its users to mainly access 2 big functionalities:

  • Accessing/searching for different resources about Creative Coding(articles, books, repositories, tutorials, etc.)
  • Add new resources related to Creative Coding

Besides these 2 main functions, the users will be able to access their own account, modify their personal data and save certain resources to their profile for ease of access.

2.3 User Classes and Characteristics

As far as user types are concerned, the application limits users by the login/signup protocol. We have 2 main categories of users: the ones with an account and the ones without an account.

A user without an account, who might be accessing the platform only once or rarely, may not want to create an account and only want to quickly reach the information they need.

On the other hand, users who choose to create an account and frequently access the platform will have the possibility to also add new resources to the database, therefore contributing to the creative coding space.

2.4 Operating Environment

This web application only requires a browser to work as intended. Because it is a web app, it is OS independent and does not require powerful hardware to work properly.

3. External Interface Requirements

3.1 User Interface

In the following sections, there will be a presentation of each page/interface of the app where the user can interact with the functionality.

Search Page

The landing page of the application is represented by the Search Page. Designed to be minimalist and easy to use, this page allows the users to access 2 main functionalities: search and login/singup. In this way, the user can branch out into the different pages of the platform by searching for a resource(which redirects the user to the Search Result page) or by loggin in(which redirects the user to the Login/Sign-up page).

Search Result Page

The Search Result page is an interface of the app that the user will reach after inputing a search string in the Search Bar. The page will return the available resources that match with the keywords in the search input. Also, on this page, the user can use filters(in the left side menu) to reduce the number of resources found to his/her preferences. Filters can include things like the prefered programming language, the type of resource, etc. Also, a common element between all pages(except Search Page) is the header, which will be kept the same across the whole app for easier access to certain functionalities.

Login/Sign-Up Page

These 2 pages- represent a basic login/signup module for our application. Here, the user will input their credentials that will be validated and if the validation is successful, the user will be redirected to the Search Page, but with new functionalities available(Account Page + Add Resources Page).

In the case of a sign up, a new user will be created and the same flow as in the case of the login will be applied.

Account Page

In the account page, a logged user can perform update actions on his personal info(Email, password, name, etc.). Also, here the user will have the possibility to see a list of saved resources, where he can access them faster than searching again for that specific resource. Again, the same common header is kept on this page as well.

Add Resource Page

In the add resource page, the user has the ability to add a new resource to the project database that can be looked up by future users. For adding a valid resource, the user will have to fill a certain amount of required data about the new resource(name, address, type, a short description). Also, the app allows for multiple "tags" to be added to the resource, by using a list.

Usecase Diagram

The following diagram acts as a representation for the flow of our application. A user will arrive at the Search Page and can choose to search for a resource with or without being logged in. After introducing the search input, the user will be redirected to the Search Result Page where he can filter further the search results or, if logged in, can save a certain resource to his profile for easier access.

If the user chooses to login or sign up, he will be redirected to the login page where his credentials will be validated. Or, if he signs up, a new user will be created with the credeantials he/she introduced. After a valid login/sign-up the user will have acces to the Search function as before, but besides, he will have access to the Account Info and Resources Functions.

After accessing the Account Page, the user will have the ability to update his personal info or see a list of saved resources associated with his profile.

When accessing the Resources Page, the user can add a new resource to the database of resources. This will be done by filling a form with the required data of a resource(name, description, category, type, etc.)

3.2 Hardware Interface

When it comes to interaction between the different functionalities of the project, they will be managed by the Nodejs server and will be represented by sending/receiving different data such as login information, search terms, filters, user data, etc. When it comes to saving resources, we will use JSON files for better structuring our database.

3.3 Software Interface

In order to build the project we decided to integrate different tools and libraries as well as combining different languages to create the features we need. To list some of the tehnologies used:

  • Node.js-building the backend server of the app.
  • HTML+CSS-building the frontend of the app.
  • JavaScript-for integrating and passing data between the components of the app.
  • MySQL-for managing the database(subjected to change in the future, if needed).

In terms of other sofware components used to develop and use our application, we just need a web browser that has developer tools integrated. The app is OS independent.

3.4 Communications Interface

In order to ensure a proper communication, the application must be run in a browser and must have access to the network. The application has to be connected to the database in order to consistently return the corresponding resources that are searched. Also, in order to ensure user safety the password will be encrypted when being sent to the backend for checking validity. Because the application is built using Node.js, it is mandatory that the backend and frontend will communicate using asynchronous functions. Also, the HTTP protocol will facilitate the communication whitin the app.

4. System Features

4.1 Keyword Matching & Filtering

Priority: High

Stimulus/Response Sequences: Input at the search bar in the main page from users.

Functional Requirements: Connection to the database.

One of the main features of our project is the Search option. Similar to a search engine, the user can input what he wants to search and the input will be parsed and separated into tokens. These tokens will then be matched with a series of keywords available in our database in order to return a search result catered to the user's needs. For example, if the user search input is "Websites in Java", the input will be broken down into 3 tokens and the first token, "Websites" will be a match with certain resources tagged with the same keyword.

Besides the keyword matching, we can narrow the search results even more by using filters. The search results page will have a side menu with filters that can be applied to the current query. For example, the user might filter by the preffered programming language(Java, Python, C/C++) or by the type of resource available(Book, article, repository, video tutorial) or even by difficulty, author, etc.

4.2 Adding Resources

Priority: High

Stimulus/Response Sequences:After a succesful login attempt, the user may press the Add button in order to add a new resource on the platform.

Functional Requirements:Connection to the database + valid login.

One other important functionality that must be present on the platform is the possibility of adding new resources. This functionality is ensured by using a dedicated page where the user will be prompted with a form where he can input the name, a short description, an address and a list of "tags" for the resource that he/she wishes to add to the actual database.

This option will only be available for logged in users, as it can be quite sensitive because it creates modifications in the database of the project.

4.3 Profile Info Update

Priority: Low

Stimulus/Response Sequences: After successfully logging in, the user can access his/her profile by clicking the user icon.

Functional Requirements: Connection to the database + valid login.

The user will be redirected to a page where he can view/edit personal data and manage a list of saved/starred resources that can be accessed quicker.
